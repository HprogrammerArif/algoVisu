# Database Schema — QuantumViz (Oracle, 3NF)

> Normalized relational schema for Oracle XE 21c. All SQL is Oracle SQL. This document
> describes tables, columns, constraints, relationships, the ER diagram, and the
> normalization justification. The actual DDL lives in `backend/db/migrations/`.

---

## 1. Conventions

- **Naming:** plural `snake_case` table names; `snake_case` columns; PK named `<entity>_id`.
- **Primary keys:** `NUMBER GENERATED BY DEFAULT AS IDENTITY` (Oracle 12c+ identity columns).
- **Text:** `VARCHAR2` for short text; `CLOB` for long text (explanations, code).
- **Timestamps:** `TIMESTAMP DEFAULT CURRENT_TIMESTAMP`.
- **Enums:** modeled as `VARCHAR2` + `CHECK` constraints (Oracle has no native ENUM).
- **Referential integrity:** every FK has an explicit constraint; user-data FKs cascade on delete.
- **Indexes:** on every FK column and on lookup columns (`slug`, `email`).

## 3. Tables

### 3.1 `roles`
Lookup table for the three user roles.

| Column | Type | Constraints |
|---|---|---|
| `role_id` | NUMBER | PK, identity |
| `name` | VARCHAR2(20) | NOT NULL, UNIQUE, CHECK in (`admin`,`teacher`,`student`) |
| `description` | VARCHAR2(255) | |

### 3.2 `users`
| Column | Type | Constraints |
|---|---|---|
| `user_id` | NUMBER | PK, identity |
| `role_id` | NUMBER | NOT NULL, FK → `roles(role_id)` |
| `full_name` | VARCHAR2(120) | NOT NULL |
| `email` | VARCHAR2(180) | NOT NULL, UNIQUE |
| `password_hash` | VARCHAR2(255) | NOT NULL |
| `is_active` | NUMBER(1) | DEFAULT 1, CHECK in (0,1) |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |
| `updated_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |

Index: `ux_users_email` (UNIQUE), `ix_users_role_id`.

### 3.3 `categories`
| Column | Type | Constraints |
|---|---|---|
| `category_id` | NUMBER | PK, identity |
| `slug` | VARCHAR2(60) | NOT NULL, UNIQUE |
| `name` | VARCHAR2(80) | NOT NULL |
| `description` | VARCHAR2(400) | |
| `display_order` | NUMBER | DEFAULT 0 |

Example rows: `sorting`, `searching`, `graph`, `grid`, `dynamic-programming`, `string`, `math`, `lists`.

### 3.4 `algorithms`
The heart of the catalog.

| Column | Type | Constraints |
|---|---|---|
| `algorithm_id` | NUMBER | PK, identity |
| `category_id` | NUMBER | NOT NULL, FK → `categories(category_id)` |
| `slug` | VARCHAR2(80) | NOT NULL, UNIQUE — joins to the client-side step-generator |
| `name` | VARCHAR2(120) | NOT NULL |
| `summary` | VARCHAR2(400) | short one-liner for the list |
| `description` | CLOB | full teaching explanation |
| `visualizer_type` | VARCHAR2(20) | NOT NULL, CHECK in (`array`,`grid`,`graph`,`matrix`,`string`,`math`) |
| `difficulty` | VARCHAR2(10) | CHECK in (`easy`,`medium`,`hard`) |
| `space_complexity` | VARCHAR2(40) | e.g. `O(1)`, `O(n)` (single value — not case-dependent) |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |
| `updated_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |

Index: `ux_algorithms_slug` (UNIQUE), `ix_algorithms_category_id`.

### 3.5 `time_complexities`
Time complexity varies by case, so it is its own table (one row per case → 3NF).

| Column | Type | Constraints |
|---|---|---|
| `complexity_id` | NUMBER | PK, identity |
| `algorithm_id` | NUMBER | NOT NULL, FK → `algorithms(algorithm_id)` ON DELETE CASCADE |
| `case_type` | VARCHAR2(10) | NOT NULL, CHECK in (`best`,`average`,`worst`) |
| `big_o` | VARCHAR2(40) | NOT NULL, e.g. `O(n log n)` |
| | | UNIQUE(`algorithm_id`,`case_type`) |

> Space complexity is stored once on `algorithms` because it is typically a single value;
> time complexity is split out because it genuinely differs per case.

### 3.6 `code_snippets`
One implementation per language (multi-language ready).

| Column | Type | Constraints |
|---|---|---|
| `snippet_id` | NUMBER | PK, identity |
| `algorithm_id` | NUMBER | NOT NULL, FK → `algorithms(algorithm_id)` ON DELETE CASCADE |
| `language` | VARCHAR2(20) | NOT NULL, CHECK in (`javascript`,`python`,`cpp`,`java`,`pseudocode`) |
| `code` | CLOB | NOT NULL |
| | | UNIQUE(`algorithm_id`,`language`) |

### 3.7 `algorithm_explanations`
In-depth, structured explanation sections per algorithm (one row per section → 3NF). Powers the
"Explain" modal in the UI. Each algorithm has 5 sections: *What problem it solves*, *How it works*,
*Why & when to use it*, *Complexity intuition*, *Real-world uses*.

| Column | Type | Constraints |
|---|---|---|
| `explanation_id` | NUMBER | PK, identity |
| `algorithm_id` | NUMBER | NOT NULL, FK → `algorithms(algorithm_id)` ON DELETE CASCADE |
| `heading` | VARCHAR2(120) | NOT NULL — the section title |
| `body` | CLOB | NOT NULL — the section text |
| `display_order` | NUMBER | NOT NULL, DEFAULT 0 — section ordering |

Index: `ix_expl_algo` on `algorithm_id`.

### 3.8 `bookmarks` (users M:N algorithms)
| Column | Type | Constraints |
|---|---|---|
| `bookmark_id` | NUMBER | PK, identity |
| `user_id` | NUMBER | NOT NULL, FK → `users(user_id)` ON DELETE CASCADE |
| `algorithm_id` | NUMBER | NOT NULL, FK → `algorithms(algorithm_id)` ON DELETE CASCADE |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |
| | | UNIQUE(`user_id`,`algorithm_id`) |

### 3.9 `progress` (users M:N algorithms, with status)
| Column | Type | Constraints |
|---|---|---|
| `progress_id` | NUMBER | PK, identity |
| `user_id` | NUMBER | NOT NULL, FK → `users(user_id)` ON DELETE CASCADE |
| `algorithm_id` | NUMBER | NOT NULL, FK → `algorithms(algorithm_id)` ON DELETE CASCADE |
| `status` | VARCHAR2(15) | NOT NULL, CHECK in (`not_started`,`in_progress`,`completed`) |
| `last_viewed_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |
| `updated_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP |
| | | UNIQUE(`user_id`,`algorithm_id`) |

## 4. Optional stretch tables

### `notes` (user 1:M)
`note_id` PK · `user_id` FK · `algorithm_id` FK · `body` CLOB · `created_at` · `updated_at`.

### `tags` + `algorithm_tags` (M:N)
- `tags` — `tag_id` PK · `name` VARCHAR2(40) UNIQUE.
- `algorithm_tags` — composite PK (`algorithm_id`,`tag_id`); both FKs ON DELETE CASCADE.

Include these only if time allows; they add normalization depth (another 1:M and another
M:N) that is nice to show evaluators.

## 5. Normalization justification

| Form | Rule | How the schema complies |
|---|---|---|
| **1NF** | Atomic values, no repeating groups | Every column holds a single value. No comma-lists; multiple code languages live as separate `code_snippets` rows, not a single column. |
| **2NF** | 1NF + no partial dependency on part of a composite key | Tables use single-column surrogate PKs (identity), and the M:N join tables (`bookmarks`,`progress`) have no non-key attributes that depend on only one half of the relationship. |
| **3NF** | 2NF + no transitive dependencies | Non-key attributes depend only on their table's key. Role text is not duplicated on `users` (it references `roles`); category text is not duplicated on `algorithms` (it references `categories`); per-case time complexity is not crammed into `algorithms` (it lives in `time_complexities`). |

This removes update anomalies (rename a category in one place), insertion anomalies (add a
category before any algorithm uses it), and deletion anomalies (deleting the last algorithm
in a category does not erase the category).

## 6. Seed data (initial)

> **Seeding approach:** seeds are applied by **TypeScript seeders** (`backend/db/seeds/*.ts`,
> run via `tsx` from `db/run.ts`) using **bind variables** rather than raw `.sql` INSERTs.
> This is required for bcrypt-hashing the admin password and is far safer for CLOB code
> snippets (whose semicolons/quotes would break raw-SQL literals). All DDL stays as
> hand-written Oracle `.sql` migrations.

Seeded by `backend/db/seeds/`:

- **roles:** admin, teacher, student.
- **a default admin user** (credentials from `.env`, hashed at seed time).
- **categories:** sorting, searching, graph, grid, dynamic-programming, string, math, lists.
- **algorithms + complexities + code_snippets:** a curated set of **15** classic algorithms
  (`backend/db/seeds/catalogData.ts`) — bubble/selection/insertion/merge/quick sort,
  linear/binary search, BFS, DFS, Dijkstra, A*, 0/1 knapsack, KMP, Sieve of Eratosthenes,
  Tower of Hanoi — each with Big-O per case, space complexity, and JavaScript + pseudocode snippets.
- **algorithm_explanations:** the 5 in-depth sections per algorithm
  (`backend/db/seeds/explanationsData.ts`, seeded by `seedExplanations.ts`).

## 7. Migration files (order)

```
backend/db/migrations/
  001_create_roles.sql
  002_create_users.sql
  003_create_categories.sql
  004_create_algorithms.sql
  005_create_time_complexities.sql
  006_create_code_snippets.sql
  007_create_bookmarks.sql
  008_create_progress.sql
  009_create_algorithm_explanations.sql
backend/db/seeds/
  seedRoles.ts
  seedAdmin.ts            -- bcrypt-hashes ADMIN_PASSWORD
  seedCategories.ts       -- includes the `string` category
  catalogData.ts          -- the 15-algorithm dataset (metadata + snippets)
  seedCatalog.ts          -- inserts algorithms + complexities + code snippets
  explanationsData.ts     -- the 5 explanation sections × 15 algorithms
  seedExplanations.ts     -- inserts the explanation sections
```

`backend/db/run.ts` (run via `tsx`) applies the `.sql` migrations then the TypeScript seeders, against the
configured Oracle connection.
