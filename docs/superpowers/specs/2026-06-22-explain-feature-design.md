## 1. Purpose

Two things:

1. **In-depth "Explain" feature.** Add an **Explain** button to the visualizer's detail panel. Clicking it opens a modal with a thorough, structured discussion of the selected algorithm: what problem it solves, how it works, why/when to use it, why its complexity is what it is, and real-world uses.
2. **Cleanup.** Remove the unused algorithm code (the ~24 fake "stub" generators and the real-but-unselected generators) left in `frontend/js/algorithms.js` after the catalog was curated to 15. **Documentation (`docs/`) is not touched.**

## 2. Locked decisions

| Decision | Choice |
|---|---|
| Explanation content source | **Stored in Oracle** (DB = "what to teach"), served via the existing detail API. No AI generation, no static frontend copy. |
| Presentation | **Modal overlay** opened by an Explain button; closes on X / Esc / backdrop click. |
| Section structure | **5 fixed sections per algorithm** (below). |
| Cleanup | Remove generators with no DB counterpart from `algorithms.js`; keep the 15 used by `ALGO_REGISTRY`. Keep all docs. |

## 3. The 5 explanation sections (per algorithm)

Every one of the 15 curated algorithms gets these five, in this order:

1. **What problem it solves**
2. **How it works** (the mechanism, step by step)
3. **Why & when to use it** (advantages, trade-offs, when *not* to)
4. **Complexity intuition** (why its best/avg/worst Big-O is what it is)
5. **Real-world uses**

15 algorithms Ă— 5 = 75 authored sections, each a substantive short paragraph.

## 4. Data model (3NF â€” mirrors `time_complexities` / `code_snippets`)

New migration `backend/db/migrations/009_create_algorithm_explanations.sql`:

```sql
CREATE TABLE algorithm_explanations (
  explanation_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  algorithm_id   NUMBER NOT NULL,
  heading        VARCHAR2(120) NOT NULL,
  body           CLOB NOT NULL,
  display_order  NUMBER DEFAULT 0 NOT NULL,
  CONSTRAINT fk_expl_algo FOREIGN KEY (algorithm_id)
    REFERENCES algorithms(algorithm_id) ON DELETE CASCADE
)
/
CREATE INDEX ix_expl_algo ON algorithm_explanations(algorithm_id)
/
```

`run.ts` adds `algorithm_explanations` to `TABLES_REVERSE` (dropped before `algorithms`).

## 5. Content storage & seeding

- **`backend/db/seeds/explanationsData.ts`** â€” exports `EXPLANATIONS: { slug: string; sections: { heading: string; body: string }[] }[]`, one entry per curated slug, each with the 5 sections. Kept separate from `catalogData.ts` (single responsibility; avoids bloating the catalog metadata file).
- **`backend/db/seeds/seedExplanations.ts`** â€” for each entry, resolves `algorithm_id` by slug; if that algorithm currently has **zero** explanation rows, inserts its sections with `display_order = 0..n`. **Idempotent and independent of `seedCatalog`'s skip logic** (the 15 algorithms already exist, so explanations must seed regardless of whether the algorithm row is new).
- Wired into `run.ts` `seed()` after `seedCatalog`.

## 8. Cleanup of `frontend/js/algorithms.js`

- **Keep** the generators referenced by `ALGO_REGISTRY`: the `addAlgo` definitions for `bubble, selection, insertion, merge, quick, linearsearch, binarysearch, bfs, astar, knapsack, kmp, sieve, hanoi`, plus the standalone `graphDFS` and `graphDijkstra`, plus the `addAlgo` helper, `ALGO_DATABASE`, and `ALGO_REGISTRY`.
- **Remove** everything with no DB counterpart: all `STUB_*` arrays and their `forEach(addAlgoâ€¦)` blocks (`STUB_ALGO_COGNITIVE, STUB_SEARCHES, STUB_STRINGS, STUB_LISTS, STUB_STACKS, STUB_TREES, STUB_GRAPHS, STUB_GRIDS, STUB_RECURSION, STUB_DP, STUB_MATH`) and the unused real generators (`quickhoare, heap, shell, cocktail, gnome, counting, oddeven, cycle, bogo, twosum, kadane, dutchflag, bstinsert, bfsgrid, gameoflife, gcd, customsort`).

- After removal, `ALGO_REGISTRY` must still resolve all 15 entries (verified by re-running the Node registry check).
- **Do not** remove or alter anything under `docs/`.

## 11. Risks

- **Re-seed gotcha:** explanations must seed even though the algorithms already exist â€” handled by `seedExplanations` keying on slug + "zero existing rows," not on `seedCatalog`'s insert.
- **Migration 009 on an existing DB:** `db:migrate`/`db:setup` drops + recreates all tables (wipes user data); for an incremental add, run the single new migration statement or `db:setup` in dev. Acceptable for this dev project.
- **Cleanup regressions:** removing a generator still referenced by `ALGO_REGISTRY` would break selection (caught by the Node registry check + manual pass).
